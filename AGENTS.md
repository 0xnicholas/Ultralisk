# AGENTS.md

本文件为 coding agent 提供指导,用于在部署开源大模型对外 API 服务时,实现生产环境必需的工程基础设施。这些模块与具体使用的模型、硬件无关,应作为独立于业务逻辑的基础层优先实现。

> 📌 具体技术选型与落地说明见 [docs/tech-stack.md](docs/tech-stack.md)(用什么做、为什么、以及落地坑)。本文件专注于"要做什么"(需求与验收标准)。

## 项目背景

- 目标:将开源大模型(通过 vLLM / TGI / SGLang 等推理引擎)封装为对外提供的生产级 API 服务
- 当前阶段:模型和硬件规模待定,需要先搭建可复用的工程基础设施
- 原则:以下四个模块应作为独立组件实现,与推理引擎解耦,便于后续更换模型或扩容而不影响这层逻辑

---

## 模块一:鉴权与限流(Auth & Rate Limiting)

### 目标
防止服务被未授权访问、恶意刷量或异常流量打垮。

### 要实现的功能
- [x] API Key 生成、分发、吊销机制(建议存储 key 的哈希值,不存明文)
- [x] 每个请求校验 API Key 合法性,拒绝无效/过期 key
- [x] 基于 QPS 的限流(建议用令牌桶或滑动窗口算法)
- [x] 基于 token 用量的配额管理(按天/按月累计,超限拒绝或降级)
- [x] 分级配额:支持为不同调用方设置不同的限流/配额策略
- [x] 超限时返回明确的错误码(如 HTTP 429)和 `Retry-After` 头,而不是静默失败

### 建议实现位置
放在 API 网关层(如 Nginx / Kong / Envoy / 自建中间件),而非业务代码内部,保证与推理逻辑解耦。

### 验收标准
- 无 Key 或错误 Key 的请求被拒绝,返回 401/403
- 超过限流阈值的请求被正确拒绝,返回 429
- 限流/配额规则可通过配置文件或管理接口动态调整,无需重启服务

## 模块二:监控(Observability / Metrics)

### 目标
实时掌握服务健康度和性能,为扩缩容、模型选型决策提供数据支撑。

### 必须采集的核心指标

| 指标 | 说明 |
|---|---|
| TTFT (Time to First Token) | 请求发出到首个 token 返回的耗时 |
| TPOT (Time per Output Token) | 后续每个 token 的生成间隔 |
| Throughput | 每秒处理的 token 数 |
| GPU 利用率 / 显存占用 | 资源使用情况 |
| 请求成功率 / 错误率 | 按错误类型细分(超时、限流拒绝、模型异常等) |
| 并发请求数 / 排队长度 | 用于判断是否需要扩容 |

### 要实现的功能
- [x] 埋点采集上述指标,暴露为 Prometheus 可抓取的 `/metrics` 端点
- [x] 关键路径(请求接收、排队、推理、返回)分别打点,便于定位瓶颈
- [x] 接入 Grafana 或等效可视化面板,搭建基础 Dashboard
- [x] 设置基础告警规则(如错误率 > 阈值、GPU 显存 > 90%、TTFT 异常升高)

### 验收标准
- Dashboard 能实时展示上述核心指标
- 人为制造高延迟/错误请求时,监控数据能正确反映

## 模块三:内容安全(Content Safety)

### 目标
防止有害输入/输出内容带来合规和法律风险。

### 要实现的功能

**输入侧(Prompt Injection 防护)**
- [x] 检测并拦截试图让模型泄露系统提示词的输入
- [x] 检测异常长度/异常重复模式的输入(可能是攻击性 payload)
- [x] 对用户输入做基础的格式校验和长度限制

**输出侧(内容审核)**
- [x] 对模型输出做敏感词/违规内容过滤,先接入规则引擎或第三方审核 API(如 Llama Guard 或等效开源方案),不要求一开始自建
- [x] 命中审核规则的输出应被拦截或替换为安全提示,而非直接返回给用户
- [x] 记录审核命中日志,便于后续迭代规则

### 建议实现方式
- 作为独立的旁路服务/中间件实现,输入审核在请求进入推理引擎前执行,输出审核在推理结果返回用户前执行
- 优先用现成方案(规则引擎 + 第三方 API),先跑通闭环,不追求一步到位自建审核模型

### 验收标准
- 已知的越狱类 prompt 测试集能被正确拦截或标记
- 违规输出内容能被过滤,不会直接透传给用户

## 模块四:日志与可观测性(Logging)

### 目标
支持线上问题排查、成本分析、审计追溯。

### 要记录的字段(每次请求)
- [x] `request_id`(全局唯一,用于全链路追踪)
- [x] 调用方标识(API Key ID,而非明文 key)
- [x] 输入内容(schema 已定义,默认不写入日志;需配合下方隐私策略启用)
- [x] 输出内容(schema 已定义,默认不写入日志;需配合下方隐私策略启用)
- [x] 请求耗时(拆分:推理时间 / 总耗时)
- [x] 使用的模型版本/名称
- [x] token 消耗量(输入 token 数、输出 token 数,用于成本核算)
- [x] 错误信息(若有),包括错误类型和堆栈

### 注意事项(隐私合规)
- [ ] 明确日志保留周期(如 30/90 天),到期自动清理 — ⚠️ 策略待确认,需与需求方/法务沟通
- [ ] 明确哪些字段需要脱敏(如涉及 PII 的用户输入) — ⚠️ 策略待确认
- [ ] 明确日志访问权限,避免未授权人员查看用户请求内容 — ⚠️ 策略待确认
- [ ] 上述策略应在实现前与需求方/法务确认,不要自行假设 — 🚩 阻塞项

### 验收标准
- 任意一次请求可通过 `request_id` 完整还原全链路耗时和结果
- 日志清理任务按策略自动执行,可验证

## 实现顺序建议

这四个模块相互独立,可并行开发,但建议按以下优先级落地(不依赖模型/硬件是否最终确定):

1. **API 网关骨架**(鉴权 + 限流)——哪怕后端只是一个 mock 推理服务,也可以先把这层跑通
2. **日志字段规范**——尽早定义好 schema,避免后期大改
3. **监控埋点**——先接入基础指标,后续可持续补充
4. **内容安全**——先用现成规则/第三方 API 顶上,后续迭代

## 非目标(暂不实现)

- 自建内容审核模型(先用现成方案)
- 复杂的多云容灾方案(等流量规模明确后再评估)
- 精细化的按用户计费系统(先做配额管理,计费系统可后续接入)

## 给 agent 的额外说明

- 以上四个模块应各自成为独立可测试的组件,不要耦合进具体推理引擎(vLLM/TGI/SGLang)的代码中,以便后续更换推理引擎或模型时无需重写这层
- 优先使用成熟开源组件(如 Prometheus/Grafana、现成的 rate limiter 库),避免重复造轮子
- 每个模块完成后,应有对应的验收测试(见各模块"验收标准"),而非仅凭代码审查判断完成度
