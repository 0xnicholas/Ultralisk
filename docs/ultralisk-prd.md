## 一句话定义

> **Ultralisk = 精简自用版 Together AI + 各种 GPU 利用率优化**
>
> 在自建数据中心上，部署 3-5 个开源大模型，作为 TokenCamp 的内部推理 Provider 平台。专注把 GPU 跑满，专注做好 OpenAI 兼容 API，不做用户系统、不做计费、不管硬件。

### 计划支持的模型（3-5 个）

| 模型 | 定位 | 何时上 |
|------|------|--------|
| **Llama 3.1 8B Instruct** | 主力小模型，轻量任务、高 QPS | Phase 1（MVP）|
| **Llama 3.1 70B Instruct** | 主力大模型，复杂任务 | Phase 1（MVP）|
| **Qwen 2.5 72B Instruct** | 中文场景主力 | Phase 2 |
| **Qwen 2.5 Coder 32B** 或 **DeepSeek Coder V2** | 代码专用 | Phase 2 |
| **BGE-M3** 或 **Qwen3-Embedding** | Embedding 服务（待定）| Phase 2+ |

> 具体选型可根据 TokenCamp 实际用户场景调整。模型清单是**目标**，不是承诺。

## 2. 背景与目标

### 2.1 为什么自建

- **战略**：不被 OpenAI/Anthropic 等外部 API 卡脖子（限流、价格、政策）
- **数据**：TokenCamp 用户的对话数据可走自建（合规、隐私）
- **成本**：规模化后，自建 GPU 比 Together AI 等便宜 30-50%
- **能力沉淀**：团队积累 AI Infra 实战经验

### 2.2 业务目标（不是商业目标）

- **目标 1**：替代 Together AI / Fireworks 在 TokenCamp 体系内的角色
- **目标 2**：用更低的成本服务同样的开源模型流量
- **目标 3**：把 GPU 利用率提升到行业领先水平（目标值待定，见待讨论章节）

### 2.3 非目标（明确不做什么）

- ❌ 不做用户系统、账户、登录
- ❌ 不做账单、发票、付费
- ❌ 不做多租户隔离
- ❌ 不做客户门户、用户控制台
- ❌ 不做硬件管理（BMC/IPMI/固件）
- ❌ 不做网络管理（InfiniBand 子网配置等）
- ❌ 不做 TokenCamp 已有的功能（用户计费、限流、压缩、PII 脱敏、加密、Fusion）
- ❌ 不对外服务（不卖算力、不接外部用户）

## 3. 产品定位

### 3.1 角色

**Together AI / Fireworks AI 的自托管版本**——但精简到只剩"推理 + 优化 + API"。

```
┌────────────────────────────────────┐
│  硬件层（不在 Ultralisk 范围）     │
│  - GPU 服务器、网络、液冷、存储    │
└──────────────┬─────────────────────┘
               ↓
┌────────────────────────────────────┐
│  ★ Ultralisk（独占整个数据中心）  │
│  - 部署开源模型                    │
│  - 管理（监控/调度/优化）          │
│  - 暴露 OpenAI 兼容 API            │
└──────────────┬─────────────────────┘
               ↓ OpenAI 兼容 API（只对内）
┌────────────────────────────────────┐
│  TokenCamp（用户端，不属于本项目）│
│  - 当 Ultralisk 是 OpenAI 一样调   │
│  - 自己处理用户侧的事              │
└────────────────────────────────────┘
```

### 3.2 与 Together AI / Fireworks 的差异

| 维度 | Together AI / Fireworks | Ultralisk |
|------|------------------------|-----------|
| **客户** | 任何企业 | **只有 TokenCamp** |
| **模型** | 50+ | 3-5 |
| **规模** | 公有云，多租户 | **单租户，独占数据中心** |
| **计费** | 按 token 收费 | **不收费** |
| **数据合规** | 数据出客户机房 | **数据全程在自有数据中心** |
| **优化目标** | 通用优化（吞吐/成本/质量）| **聚焦 GPU 利用率** |

### 3.3 与 TokenCamp 的边界

| TokenCamp 做 | Ultralisk 做 |
|-------------|-------------|
| 用户管理 | 推理服务 |
| 用户计费 | 模型管理 |
| 用户限流 | 资源调度 |
| Prompt 压缩 | 推理优化 |
| PII 脱敏 | 故障自愈 |
| 加密 | 监控告警 |
| Fusion 多模型融合 | 内部计量（成本归因）|
| 多 Provider 路由 | 暴露 OpenAI 兼容 API |

**核心契约**：TokenCamp 把 Ultralisk 当作 OpenAI 调用，**TokenCamp 不需要知道 Ultralisk 内部实现**。

## 4. 用户与场景

### 4.1 唯一上游：TokenCamp

**不是终端用户**。TokenCamp 是唯一直接调用者，终端用户是 TokenCamp 的客户。

| **Embedding / 检索** | 批量、计算密集 | Embedding 模型 |
| **代码生成** | 中等 QPS、长输出 | 代码专用模型（可选）|
| **中文场景** | 看 TokenCamp 用户分布 | Qwen 系列（可选）|

### 4.3 用户故事

| 编号 | 角色 | 故事 |
|------|------|------|
| **US-1** | TokenCamp（上游）| 我能把 Llama 3 70B 当作 OpenAI 模型一样通过 OpenAI 兼容 API 调用 |
| **US-2** | TokenCamp | 我能流式获得响应（SSE）|
| **US-3** | TokenCamp | 我能切换到不同模型（`model` 参数指定）|
| **US-4** | 平台运维（内部团队）| 我能在一个地方看到所有模型的 GPU 利用率、QPS、延迟 |

| **US-7** | 平台运维 | 我能滚动升级模型权重而不中断服务 |
| **US-8** | 平台运维 | 我能配置一个新模型并让它上线（YAML 文件改一下）|

> **角色说明**：
> - **TokenCamp**：唯一上游调用方，把 Ultralisk 当 OpenAI 用
> - **平台运维**：3-5 人的内部团队，负责 Ultralisk 部署、监控、故障处理

## 5. 核心能力

### 5.1 推理服务（核心）

**做什么**：跑开源大模型，对外提供 OpenAI 兼容 API。

**输入**：HTTP 请求（OpenAI 兼容格式）
**输出**：HTTP 响应（流式或非流式）
**约束**：
- 支持 3-5 个模型同时在线
- TTFT < 500ms（p95）
- 持续吞吐 ≥ 100 tokens/s/请求（视模型而定）

### 5.2 多模型管理

**做什么**：在一个集群里管理 3-5 个模型。

**策略**：多实例多模型 + 动态加载（Phase 2+）

**详细**：
- **小模型**（< 13B）：常驻加载，随时可用
- **大模型**（≥ 70B）：按需加载，冷启动 < 60s
- 模型定义在 YAML 文件里，启动时加载
- 热更新（修改 YAML → 自动加载/卸载）Phase 2+ 支持

**MVP 状态**：**MVP 阶段所有配置的模型都常驻启动**，不动态加载。Phase 2+ 引入动态加载。

### 5.3 资源调度与优化

**做什么**：把 GPU 跑满，降低单位 token 成本。

**详细**（优化部分单独讨论，见优化层章节）：
- 多模型共享 GPU（MIG 切分、时分复用）
- 动态扩缩容（流量低时缩）
- 预加载（基于流量预测）

### 5.4 故障自愈

**做什么**：节点故障、网络抖动、模型崩溃时自动恢复。

**详细**：
- 节点心跳检测
- 失败任务自动重试
- GPU 故障时自动迁移到健康节点
- 服务降级（部分模型不可用时返回明确错误）

**MVP 状态**：**不在 MVP**。MVP 阶段故障由人工处理（K8s 重启 + 手动告警），Phase 3 才做自动迁移。

### 5.5 监控与计量

**做什么**：让运维看清楚一切，让决策有数据支撑。

**详细**：
- **业务指标**：QPS、TTFT、TPOT、token 吞吐、错误率
- **资源指标**：GPU 利用率、显存占用、网络吞吐
- **成本归因**：每模型每小时的 token 数 + GPU 成本
- **告警**：节点离线、GPU OOM、QPS 异常

### 5.6 接口（OpenAI 兼容）

**做什么**：让 TokenCamp 无缝调用。

**详细**：
- 端点：`/v1/chat/completions`、`/v1/completions`、`/v1/embeddings`、`/v1/models`
- 鉴权：服务密钥（`Authorization: Bearer ...`）
- 流式：SSE（Server-Sent Events）
- 模型名：用户友好名（`llama-3-70b`），内部映射到实际模型权重

## 6. 优化层（核心差异化）

> **优化层是 Ultralisk 的核心差异化**——不是基础能力，是**让自建比 Together AI 更值**的关键。

借鉴 Chamber（[usechamber.io](https://www.usechamber.io)）的产品思路，按价值分 3 层：

```
┌──────────────────────────────────────────┐
│  L3 智能体 (AIOps)        [P2，未来]    │
│  - 自动根因分析、自动迁移               │
│  - Slack bot 问答                       │
├──────────────────────────────────────────┤
│  L2 智能调度              [P1，差异化]  │
│  - 多模型共享 GPU                       │
│  - 容量保证 + 弹性抢占                  │
│  - 把利用率从 30% 提到 60%+             │
├──────────────────────────────────────────┤
│  L1 监控 + 计量           [P0，必须]    │
│  - 业务 + 资源 + 成本三视图             │
│  - Grafana 仪表盘                       │
│  - 成本归因                             │
└──────────────────────────────────────────┘
```

### 6.1 L1：监控 + 计量（P0，必须有）

**目标**：让运维"看见"一切——业务、资源、成本三个维度。

**业务指标**（来自推理引擎 + API 网关）：

| 指标 | 含义 | 来源 |
|------|------|------|
| QPS | 每秒请求数 | API 网关 |
| TTFT | 首 token 延迟 | vLLM |
| TPOT | 每输出 token 延迟 | vLLM |
| Token 吞吐 | 输入/输出 tokens/s | vLLM |
| 错误率 | 4xx/5xx 比例 | API 网关 |
| 流式比例 | 流式请求占比 | API 网关 |

**资源指标**（来自 GPU exporter + DCGM）：

| 指标 | 含义 |
|------|------|
| GPU 利用率 | SM 活跃百分比 |
| 显存占用 | 已用 / 总显存 |
| 显存碎片率 | 不可用显存比例 |
| 功耗 | GPU 实际功耗 |
| 温度 | GPU 温度 |
| 网络吞吐 | 节点间 / 节点内 |

**成本指标**（自定义计算）：

| 指标 | 含义 |
|------|------|
| 每模型 token 数 | 输入 + 输出，按小时聚合 |
| 每模型 $/M tokens | GPU 成本 / token 数 |
| 每模型 GPU 时长 | 各模型占用的 GPU·小时 |
| 闲置率 | 未利用 GPU 容量比例 |

**交付物**：
- **3 个指标维度**：业务（QPS/TTFT/吞吐/错误率）、资源（GPU 利用率/显存/功耗）、成本（$/M tokens/闲置率）
- **Grafana 仪表盘**：总览 + 按模型拆分 + 资源 + 成本 4 个视图
- **Prometheus 告警规则**：节点离线、GPU OOM、QPS 异常
- **月度报告自动生成**（Phase 2）：token 消耗、成本、利用率趋势

**MVP 状态**：**部分实现**——基本业务指标 + 资源指标，**成本归因和月度报告可后置**。

### 6.2 L2：智能调度（P1，差异化关键）

**目标**：把 GPU 利用率从"自然跑"的 30% 提到 60%+。

**Chamber 风格的调度模型**（两阶段）：

```
┌─────────────────────────────────────────────┐
│  Reserved（预留容量）                        │
│  - 生产模型有保底 GPU 数量                   │
│  - 不会被抢占                                 │
│  - 例：Llama 70B 预留 N 张卡                 │
├─────────────────────────────────────────────┤
│  Elastic（弹性容量）                          │
│  - 共享 GPU 池                               │
│  - 低优先级流量可抢占                          │
│  - 忙时自动让出给 Reserved                    │
│  - 例：实验模型、批量任务、Embedding         │
└─────────────────────────────────────────────┘
```

**Ultralisk 单租户的实际情况**：

- **没有"实验流量"**——TokenCamp 的所有流量都是生产
- **但有多模型**——Llama 70B、8B、Qwen 等是不同模型
- **多模型本身就是"分层"**——小模型弹性、大模型保底

**Ultralisk 的调度策略**（简化）：

| 策略 | 说明 |
|------|------|
| **大模型保底** | Llama 70B 永远有 ≥ N 张 GPU 可用 |
| **小模型共享** | Llama 8B 跟其他小模型共享 GPU 池 |
| **按需加载** | 大模型流量高峰时临时加载备用实例 |
| **超时回收** | 备用实例空闲 N 分钟后自动卸载 |

**多模型共享 GPU 的实现选项**：

| 方案 | 复杂度 | 利用率提升 | 风险 |
|------|--------|----------|------|
| **MIG 切分** | 中 | 中 | 仅 A100/H100 支持，硬隔离 |
| **MPS** | 中 | 中 | 仅软件隔离，性能有损 |
| **时分复用** | 低 | 中 | 单卡同时只跑一个模型实例 |
| **不共享** | 0 | 0 | 每个模型独立占卡 |

**Ultralisk 推荐**：MVP 用**时分复用**（最简单），Phase 3 引入 **MIG**（A100/H100）。

**交付物**：
- 调度器（基于 Volcano / KAI Scheduler）
- 模型优先级配置（YAML）
- 自动扩缩容（基于 QPS、队列长度）
- 容量保证（生产模型保底）

**MVP 状态**：**不在 MVP**——MVP 只用时分复用（每模型占满自己的卡），Phase 2 再做共享。

### 6.3 L3：智能体 (AIOps)（P2，未来扩展）

**目标**：减少人工运维介入。

**Chamber 风格的智能体能力**：

| 能力 | 说明 | 价值 |
|------|------|------|
| **自动根因分析** | "为什么延迟升高"自动诊断 | 高 |
| **自动迁移** | 节点故障时自动迁移工作负载 | 高 |
| **自动扩缩容** | 流量变化时自动调容量 | 中 |
| **Slack bot** | 问"GPU 跑得怎么样" | 低（团队小） |
| **预测性告警** | 提前预测 OOM、节点故障 | 中 |

**Ultralisk 的智能体设计**：

- **不重复造轮子**——Kubernetes + KServe 已有基础的自动恢复
- **只在 K8s 不够用时介入**——例如"两个模型同时 OOM"这种复杂场景
- **MVP 不做**——团队 3-5 人手动运维可以接受

**交付物**（Phase 3）：
- Slack/CLI bot（自然语言查询集群状态）
- 自动迁移触发器（节点故障时）
- 异常检测（流量模式异常告警）

### 6.4 优化层总结

| 层 | 优先级 | 何时做 | 价值 |
|----|--------|--------|------|
| **L1 监控 + 计量** | P0 | MVP（基础）+ Phase 2（成本归因）| 看见问题、归因成本 |
| **L2 智能调度** | P1 | Phase 2-3 | 利用率 30% → 60%+ |
| **L3 智能体** | P2 | Phase 3-4 | 减少人工 |

### 6.5 推理引擎层优化（Chamber 不做，Ultralisk 要做）

Chamber 是基础设施层优化（调度、监控、迁移），**不做推理引擎优化**。但 Together AI 的核心优势之一就是推理引擎优化：

| 优化 | 价值 | 何时做 |
|------|------|--------|
| **INT4 量化（AWQ）** | 节省 50%+ 显存，吞吐 +30% | MVP（杀手优化） |
| **Continuous batching** | 吞吐 + 2-4x | MVP（vLLM 默认） |
| **Prefix caching** | 重复 prompt 加速 5-10x | Phase 2 |
| **Speculative decoding** | 输出加速 2-3x | Phase 4 |
| **Prefill-Decode 分离** | 延迟降低 30-50% | Phase 4 |
| **KV cache 优化** | 长上下文加速 | Phase 3 |
| **自定义 CUDA kernel** | 极致性能 | 视情况 |

**Ultralisk 的定位**：
- **L1-L3**（基础设施优化）= 跟 Chamber 学的
- **推理引擎优化** = 跟 Together AI 学的
- **MVP 至少要有一个**——量化是性价比最高的

## 8. MVP 范围

### 8.1 MVP 定义

**MVP = 最小可被 TokenCamp 集成的版本**。

不是 Together AI 全功能，是**Together AI 的最小子集 + 一个杀手优化**。

### 8.2 MVP 必须有

| 能力 | 说明 | 对应优化层 |
|------|------|----------|
| ✅ **MVP 第一个模型：Llama 3 8B** | 先用小模型验证（1 周内可跑通） | - |
| ✅ **MVP 第二个模型：Llama 3 70B** | 小模型稳定后加（验证 2 模型并存） | - |
| ✅ **OpenAI 兼容 API** | `/v1/chat/completions`、流式 + 非流式 | - |
| ✅ **基础监控** | Prometheus + Grafana，看得到关键指标 | L1（部分） |
| ✅ **静态配置** | YAML 启动加载 | - |
| ✅ **量化（INT4）** | 节省 50%+ 显存 | 推理引擎层优化（杀手）|
| ✅ **Continuous batching** | vLLM 默认开启 | 推理引擎层优化 |
| ✅ **TokenCamp 集成验证** | TokenCamp 能调用并得到正确响应 | - |
| ✅ **内部计量** | token 数记录（不复杂，能算账即可） |

### 8.3 MVP 暂不做

- ❌ 多模型（先 2 个）
- ❌ 动态加载
- ❌ 热更新配置
- ❌ 故障自愈（手动恢复）
- ❌ 复杂调度（MIG、prefill-decode 分离）
- ❌ 流量预测
- ❌ 自动扩缩容
- ❌ 详细成本归因（只有总量）
- ❌ Embedding 服务
- ❌ 高级优化（speculative decoding 等）

### 8.4 验收标准（MVP Done = ?）

**8B 阶段验收**（Step 1 Done）：

- [ ] Llama 8B 能跑起来，TTFT < 500ms p95
- [ ] INT4 量化生效

**70B 阶段验收**（MVP 全 Done）：
- [ ] Llama 70B 能跑起来，TTFT < 1s p95
- [ ] 8B + 70B 同时在线，互不干扰
- [ ] 运维文档完整（部署、升级、排查）

### 8.5 MVP 时间

**6 周**（含 8B 验证 + 70B 验证）：
- **第 1-3 周**：8B 跑通
- **第 4-6 周**：70B 加入，验证 2 模型并存
- **总计 1.5 月**（与路线图 1-2 月对齐）

### 9.1 Phase 1: MVP（1.5 月 / 6 周）

| 步骤 | 时间 | 目标 | 验收 |
|------|------|------|------|
| **Step 1: 8B 跑通** | 第 1-3 周 | 验证整个链路 | TokenCamp 能用 8B，端到端跑通 |
| **Step 2: 70B 加入** | 第 4-6 周 | 验证 2 模型并存 | 8B + 70B 同时在线，互不干扰 |

**交付**：
- 2 个模型（8B + 70B，分步上线）
- 简单 Provider API
- 基础监控（L1 部分）
- INT4 量化
- TokenCamp 集成
- 运维文档（部署、升级、排查）

### 9.2 Phase 2: 多模型 + 调度基础（2-4 月）

- 扩展到 3-5 个模型（含 Qwen 72B + 代码专用）

- Continuous batching 参数调优
- 完整成本归因（每模型 $/M tokens）

### 9.3 Phase 3: 规模 + 智能体（4-9 月）

- 多节点 GPU 集群
- 多模型 GPU 共享（MIG）
- 流量预测 + 预加载
- 故障自愈
- 自动扩缩容

### 9.4 Phase 4: 极致推理优化（9-18 月）

- Prefill-decode 分离
- Speculative decoding
- KV cache 优化
- 自定义 CUDA kernel（如必要）
- 极致 GPU 利用率（目标 80%+）

## 10. 风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| **自建比 Together AI 贵** | 中 | 规模化和利用率优化对冲 |

| **vLLM 性能不达预期** | 中 | SGLang/Triton 备选 |
| **TokenCamp 协议变更** | 低 | OpenAI 兼容是行业标准，影响小 |
| **NVIDIA 工具替代** | 中 | 紧跟生态但不依赖单一 |
